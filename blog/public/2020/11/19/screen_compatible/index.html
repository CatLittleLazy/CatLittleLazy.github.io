关于屏幕适配

基础知识部分

  • 对于了解屏幕适配我们需要先了解如下知识

    • px pixel
    • resolution
    • ppi(pixels per inch)
    • dpi(dots per inch)
    • dp dip
    • sp

  • px

    • pixel，像素，组成屏幕图像的最小独立元素
    • 构成屏幕的发光的点，是物理的
    • 脱离了屏幕尺寸之后是没有大小可言的
      • 可以将 1920 * 1080 颗像素放到一台 40 寸的小米电视机里面，也可以将同样多的像素全部塞到一台 5.5 寸的 iPhone7 Plus 手机里面去，那么对于 40 寸的电视而言，每个像素颗粒当然会大于 5.5 寸的手机的像素。

  • resolution

    • 分辨率，标识屏幕或图片所包含的像素个数，即像素尺寸
    • 如Pixel2xl的分辨率2880*1440，宽高比为18:9
      • 18:9的说法，是与之前手机屏幕的16:9，17:9进行对比，更容易直观的比较。
    • 可以简单理解为2880是高度上的所有像素点，1440为宽度上的所有点，那么Pixel2xl则共包含了2880 x 1440 = 4147200个像素点
    • 物理分辨率为硬件所支持的，逻辑(支持)分辨率为软件可达到的
      • 例如在iphone3gs的时候，逻辑分辨率是320x480，物理分辨率也是320x480，iphone4出了之后，逻辑分辨率和3gs一样是320x480，显示的内容是一样多的，但是物理分辨率变成了640x960，原来在3gs的一个像素内容，在iphone4上则填充了4个像素，因此虽然内容显示还是一样多，但是iphone4s的屏幕精细度比3gs高了2倍，像素倍率是2.
      • iphone6的逻辑分辨率是375x667，比iphone4/4s/5/5s/SE逻辑分辨率都要大，所以显示的内容更多。物理分辨率是750x1334，所以和上边谈到的机型一样，仍然是2倍的像素倍率

  • ppi与dpi

    • 均为描述关于成像密度的描述单位
    • dpi起源于纸媒时代印刷机印刷时在单位长度内的墨点密度，最早用于描述打印机性能
      • 最多能用多少个墨点来打印一寸的内容
    • ppi描述显示器单位长度内的最小成象的密度，用于描述屏幕的像素密度
      • 上面提到的印刷物以无数多的墨点来构成图像，而屏幕同样也是以一定数量的发光点来构成图像
      • 对于屏幕来说 PPI 是用于描述每英寸发光点数量的
      • 一块屏幕宽高有几寸是在生产的时候就被定好的，而宽高各能容纳下多少颗像素，也是在生产的时候就被定好的,所以我们所说的 PPI 可以说是一个物理单位。
    • 当dot为pixel时两者值相同
    • 手机ppi为屏幕对角线上的每英寸像素点个数，勾股定理得出对角线上的像素点个数，再除以手机屏幕尺寸即可得出，以iPhone6为例，ppi = (开方)(750 * 750 + 1334 * 1334) / 4.7 = 325.6
    • 同理得出Pixel2xl的ppi = (开方)(2880 * 2880 + 1440 * 1440) / 6.0 = 536.65

  • dp与dip

    • density-independent pixels无关密度像素，同一个概念
    • 是一个虚拟像素单位，其具体值与手机ppi有关

  • sp

    • Scale-independent Pixels无关缩放像素
    • 用于设置文字字体大小，默认与dp相同，不过会根据用户的首选文本大小来调整大小

AutoLayout适配方案思路

  • AutoLayout地址

  • 在AutoLayoutConfig.getInstance().init(context)中有如下方法

    • 获取了设计稿的分辨率以及手机的分辨率

  • 在该适配方案中所有的Activity都继承自AutoLayoutActivity

  • 以AutoLinearLayout为例

    • AutoLinearLayout继承自LinearLayout
    • 重写onMeasure()方法通过AutoLayoutInfo中持有的AutoAttr对子View的属性进行适配

  • 继续进行apply(View)方法

    • AutoAttr为抽象类
    • 在apply方法中对View的属性值进行适配计算
    • 计算过方式即通过AutoUtils工具类的getPercentWidthSizeBigger(pxValue)等方法进行计算
    • 计算规则即是按照实际稿与实际值的比例进行设置
    • 最后通过exec(vew,val)对适配后的值进行设置

  • AutoAttr为抽象类，以Height为例，即HeightAttr

    • HeightAttr继承自AutoAttr
    • 在exec(View,int)方法中对ViewGroup.LayoutParams.height进行了设置，从而完成高度的适配

  • 一个问题

    • 支持16:9的屏幕，目前已经不维护了，对于全面屏手机不再适用
    • 如计稿为1920 * 1080，而Pixel2xl屏幕分辨率为2880 * 1440
    • 此时设计稿的高宽比例为16:9;而实际屏幕高宽比为18:9
    • 由于该适配思路是按照百分比缩放进行
    • 所以在实际机型与设计稿比例相差较多情况下，继续按照百分比缩放的话将会界面适配时失败
    • 如上例中将会导致高度变高，若在设计稿中为正方形，在Pixel2xl上将呈现一个长方形
    • 粗糙的修改方案为调整init逻辑，添加当前宽高比与设计稿宽高比的对比，以宽为基准，对高进行重新设置
